Premortem — the go-real transition

LLM load-balancer / task-aware router, the day after it goes live on Azure with real Foundry creds, real dev prompts, and a reachable endpoint.

8 failure modes investigated in parallel · 2 confirmed in code · frame: "it's 6 months out and this has failed — why?"

Synthesis — read this first

Most likely failure

#2 · Real personal/customer data reaches Foundry

Near-certain the instant real traffic flows — because nothing technical has to break. Devs point coding agents at real client repos; the agent slurps surrounding context (customer names, secrets, a prod DB row in a stack trace) and streams it through the one endpoint to Foundry. The system routes it perfectly. The governance rule ("no customer data on a real-model path") was never code — it lived in the discipline of using synthetic test prompts. The "no prompt content stored" design then makes the blast radius unknowable: you can't even prove what leaked.

Most dangerous failure

#3 · The governance guard fails open under enforce confirmed in code

Lower-visibility, catastrophic, and a real code path today. _policy_stateless (e2e/obs_callback.py:561) treats a missing/empty key_models as "unrestricted" and keeps every candidate, Foundry included. key_models comes from getattr(user_api_key_dict, "models", None) — which the code itself warns varies across the three inbound protocols. On the Codex/Responses path the allowlist can arrive empty → a restricted key is silently treated as unrestricted → the enforce rewrite sends it to Foundry, which LiteLLM never re-checks. No backstop, by design. Detection came from the Foundry bill, not the gateway.

The hidden assumption

"A rule we wrote down is a control we enforce."

The one thing so obvious it stopped looking like an assumption. Every top failure is a documented intention standing in for an enforced mechanism:

  • Governance rule → a sentence in CLAUDE.md, not content inspection (#2)
  • Dashboard "localhost only" → a docstring, while the code binds 0.0.0.0 (#6, confirmed)
  • Auto-merge tripwire → prose the CI never reads, not a technical gate (#7)
  • Enforce "sole guard" → holds only until the "degrade to no block on error" path quietly removes it (#3)

Underneath it: "green against the synthetic harness means safe in prod." The harness is 100% mock backends, mock Foundry, synthetic prompts, one container, localhost. It is green precisely because it tests the one configuration where the go-real threats can't exist.

The revised plan

  1. Flip governance to fail CLOSED. Missing/empty/unloadable key_models ⇒ most-restrictive (local-only) candidate set. "Unrestricted" must require an explicit wildcard, never an absence.
  2. Make Foundry opt-in, default-deny per key until DISCO signs off on Foundry for a named data class — and get that sign-off against the control that actually exists (backend-gating, no DLP), not an assumed one.
  3. DASH_HOST=127.0.0.1 + auth (Azure Easy Auth / shared token). Startup refuses a 0.0.0.0 bind unless auth is set. :9300 never on public ingress.
  4. Postgres pin store before >1 replica (the deferred docs/12 §3 call) — or pin replicas to 1 + scale-to-zero off, explicitly, with the migration as the documented scale gate.
  5. Turn the tripwire into a CI gate that fails the merge if a deploy-from-main workflow exists. Decide the release model before wiring any deploy.
  6. Push-based spend alert (SpendLogs → alert at X%/hour), real per-key budgets from real pricing, and verify prompt-caching on the real route before trusting the escalation cost model.
  7. Real-Foundry soak test: separate capacity-429 from rate-429 (don't fan out retries on capacity), re-tune cooldowns, cap fallback fan-out, map auth/quota errors to honest client errors.

Pre-launch checklist

  • Governance fails closed — a test proves a key with no/empty/unloadable allowlist cannot reach Foundry under enforce, on all three inbound protocols (esp. Codex/Responses). Until green, do not flip enforce in prod.
  • DISCO sign-off matches reality — in writing, which data classes may touch Foundry; Foundry default-deny per key.
  • Dashboard not public — DASH_HOST bound + authed; verified with an external port scan of the deployed endpoint.
  • Pin store decided — Postgres-backed, or replicas=1 + scale-to-zero off, documented as the scale gate.
  • No deploy-from-main until the release model is decided AND a technical gate enforces the tripwire; spend alert live before real creds; caching verified or escalation re-priced.

The eight failure modes

FAILURE 02

Real customer data reaches Foundry

Likelihood: very highSeverity: critical

Devs pointed Claude Code at a live client repo; the agent slurped a config with a customer name, a .env fragment, a prod DB row in a stack trace, and streamed it to Foundry. The gateway routed perfectly and logged metadata only — so nothing was inspected, ever. Surfaced 3 months later when a Sulava client's vendor-data-flow review asked "does the shared gateway send our data to Foundry?" Yes, thousands of times. DISCO's sign-off (granted assuming a control existed) was voided; breach review across all five orgs.

Underlying assumption

That a governance rule enforced by test discipline holds once real, uncontrolled human+agent traffic replaces the synthetic harness.

Early warning signs
  • No one can name a single real dev prompt that ever flowed — the control has never been tested against its actual threat.
  • DISCO approval references "content controls / data inspection" that exist nowhere in the code; the filter only gates backends by key.
FAILURE 03

Governance filter fails open under enforce

Likelihood: highSeverity: criticalconfirmed in code

_policy_stateless only builds an allowlist when key_models is a non-empty list of real names; otherwise allow=None → "governance: key unrestricted" → every candidate kept. On the Responses/Codex bridge the key allowlist came back empty, so local-only keys were silently unrestricted. The agent_capable gate + unhealthy local workbenches left only Foundry surviving; the rewrite pointed there; LiteLLM never re-checks a rewritten model. Ran for weeks — the one governance e2e used a populated allowlist, so the broken path had no coverage. Caught by Foundry billing.

Underlying assumption

That a restricted key's allowlist always reaches the policy as a populated list — so "no allowlist" safely means "unrestricted" rather than "failed to load."

Early warning signs
  • ROUTING_RECORD lines where a known-restricted key_alias shows reason: "governance: key unrestricted" or a chosen on the foundry tier — visible in shadow, before enforce flips.
  • Delivered records: local-only key_alias with a Foundry served_model.
FAILURE 07

Auto-merge tripwire missed → ships to live

Likelihood: mediumSeverity: critical

The Azure deploy got wired to push-on-main the obvious way. Nobody flipped the CLAUDE.md tripwire from prose to reality — it lives in a markdown file the pipeline never reads, and the release-model decision stayed "fine as-is." An overnight goal run branched, passed the synthetic gate (no real backends), squash-merged per contract, and the change deployed to the live endpoint. A routing/governance tweak that was green against mocks weakened the sole guard on real traffic, with nobody watching.

Underlying assumption

A safety rule written as prose will be honored at the exact moment it matters, even by an unattended agent that never re-reads it against changed infra.

Early warning signs
  • A deploy workflow triggering on merge to main exists while GOALS.md still lists the release model as undecided.
  • Auto-merged PRs touching routing/enforce/governance whose only evidence is the synthetic gate — zero prod smoke between merge and live.
FAILURE 06

Unauthenticated dashboard exposed

Likelihood: highSeverity: highconfirmed in code

Smoking gun: e2e/dashboard.py:1810 defaults DASH_HOST to 0.0.0.0, not localhost — the docstring says localhost, but a comment isn't a control. The gateway's :443 ingress config got copied to :9300 "temporarily" during a rushed cutover. An internet-wide scanner indexed the open port; GET /api/records returns clean JSON to anyone. Leaked: user_ids/key aliases → models, session patterns, per-team spend, and the fleet registry with every workbench's api_base + health — a live internal attack map. "Metadata only" was the fatal comfort.

Underlying assumption

That a tool's intended context ("localhost test daemon") constrains its actual one — a documented convention substituting for an enforced bind + auth.

Early warning signs
  • A grep DASH_HOST shows 0.0.0.0 shipped in code while every doc claims localhost-only.
  • The exposure model stayed an open Needs-a-human item after the go-live date was set.
FAILURE 05

Cost blowout

Likelihood: highSeverity: high

Dominant driver: the escalation hop re-ingests the whole (already huge) agentic transcript uncached into premium Foundry, repeatedly, within one session. The "route cheap, escalate rarely" premise assumed caching absorbed the re-reads — but caching on the Foundry route was never verified and wasn't active. A stuck Codex loop times several concurrent devs compounded it. Synthetic budget tests (max_budget:0→400, rpm:1→429) proved the plumbing fires, not that the numbers were sane. The dashboard is a pull, not a push — first signal was the Azure invoice.

Underlying assumption

That enforcement plumbing passing synthetic extremes meant the budget numbers were calibrated for real agentic + escalation volume — and that caching made re-ingestion cheap.

Early warning signs
  • SpendLogs shows a session's cost jumping stepwise on each escalation; input-token counts never drop across turns.
  • Foundry premium input tokens climbing while cache-hit rate sits near zero.
FAILURE 08

Real Foundry breaks mock-tuned assumptions

Likelihood: highSeverity: high

Peak-hour 429s weren't the rate-limit 429s mockd threw — they were regional capacity-429s. The fallback treated them identically and, since the next backend was the same exhausted region, the chain fanned out instead of failing over — multiplying load exactly when Foundry had none. Cooldowns tuned against a millisecond-recovery mock were far too short; backends flapped every heartbeat. Big-model streaming ran single-digit tok/s (risk 3). Caching didn't apply (risk 5), so cost ran 3–5×. Token expiry surfaced as opaque 5xx.

Underlying assumption

A deterministic mock's fault taxonomy and timing are a valid proxy for a real capacity-constrained, latency-variable, caching-uncertain cloud backend.

Early warning signs
  • Fallback-attempts-per-request > 1 on average, correlated with time-of-day — the chain is fanning out, not failing over.
  • Cost-per-escalated-turn diverging upward; cached-token ratio near zero on the Foundry route.
FAILURE 01

Leaked key → free LLM / cred drain

Likelihood: mediumSeverity: high

A virtual key (not the master) escaped — into a client-shared GitHub repo via a .env variant .gitignore missed. GitHub secret-scanning doesn't flag sk- LiteLLM keys. A permissive ingress made the endpoint reachable from anywhere the key was. The per-key budget was a spend cap, not a breach alarm: the attacker ran the key flat to the ceiling each cycle — "within budget" on the dashboard, so nothing screamed. Found ~5 weeks later when Foundry spend pinned at ceiling every cycle from unrecognized prompts — now a DISCO question of what an attacker sent through a Projectum-only path.

Underlying assumption

That a key in a trusted internal dev's hands stays private — so budgets/rate-limits suffice and key exposure itself needs no detection.

Early warning signs
  • A key hitting its budget ceiling at a steady max every cycle, or requests from IPs/user-agents outside the known dev fleet.
  • Any key value in a git push / CI log; the :9300 dashboard reachable off-VNet at all.
FAILURE 04

Pin store breaks across Azure replicas

Likelihood: highSeverity: medium–high

The guarded SQLite UPDATE gave real pin-once / exactly-once — within one container. Container Apps scaled to 3 replicas, and "atomic within one file" became "atomic within one of three." Session turns hit different replicas → re-pinned mid-conversation (format contract broken, context re-ingested, cost + latency spikes). Escalation was worse and silent: "exactly once per session" held per file, so a session bouncing across replicas escalated once per replica — double cost, duplicate side-effects. Scale-to-zero wiped /tmp, re-pinning returning sessions. Going live on Azure was "replica time"; the deferred Postgres call was never made.

Underlying assumption

That "one container, many workers" and "many containers" are the same trust boundary for shared state.

Early warning signs
  • Escalation/pin counts exceed distinct-session counts once replicas > 1.
  • One session_id logged with conflicting pinned-tier decisions within a conversation, correlated with pod identity; pin-DB rows resetting after every scale-to-zero.
